To evaluate the effectiveness of the LoopPoint methodology in full-system simulation, we apply the complete sampling pipeline---using both the LoopPoint Analysis and PcCount Tracker modules---to the NPB benchmark suite.
We use Class~A input size with eight threads for our experiments, as larger input sizes (Class~B and above) do not complete baseline full-system simulation within a reasonable time frame.
In addition to evaluating in a full-system environment, we test the methodology across different ISAs and system configurations, as detailed in Table~\ref{tab:gem5-config}.
We use Ubuntu 24.04 as the guest operating system in all experiments.

We conduct phase analysis and checkpoint creation in gem5's functional mode.
The checkpoints of selected representative regions are then simulated in detailed mode using the configurations shown in Table~\ref{tab:gem5-config}.
We evaluate the accuracy of the LoopPoint methodology by comparing the predicted runtimes against baseline full simulations.
Figure~\ref{fig:gem5_fs_npb_A} shows the relative prediction error for the NPB Class~A benchmarks.

The Class~A input size does not produce enough regions for the methodology to build an accurate sample set.
For example, only eight total regions are discovered for \textit{cg} (from which two representatives are selected) and three for \textit{is} (from which one is selected).
With so few regions, the clustering and selection process has limited statistical power.
The prediction errors are not consistent across the x86 and Arm configurations.
Because LoopPoint constructs BBVs from machine-level basic blocks and defines region boundaries with ISA-specific (PC, count) pairs, each ISA binary produces a different set of regions, clusters, and representative samples.
As a result, using predicted runtimes to compare performance across ISAs can compound the error beyond the absolute error of each individual prediction.
This underscores the need for ISA-independent samples---the same set of representative regions used across different ISAs---to avoid the additional error introduced by independently selecting samples for each configuration.

Figure~\ref{fig:gem5_npb_A_all} also shows the speedup achieved by sampled simulation.
The speedup is calculated as the reduction in the number of instructions simulated in detailed mode compared to the full baseline.
Due to the limited number of regions discovered for Class~A benchmarks, the selected representatives constitute a large fraction of the total regions, resulting in modest speedup for this experiment.
The speedup from the LoopPoint methodology primarily depends on the number of samples required to achieve a target accuracy, rather than on the simulator itself; larger workloads with more regions would yield greater reductions.

These results demonstrate that the LoopPoint methodology can be extended to full-system simulation in gem5, but they also highlight three key limitations.
First, validating the methodology's accuracy requires completing a full baseline simulation---the very bottleneck that sampling aims to avoid.
Second, because LoopPoint's analysis is tied to a specific binary, each ISA produces a different set of representative samples, making cross-ISA comparison unreliable.
Third, interval analysis via functional simulation is time-consuming; in Section~\ref{sec:comparison_with_functional_simulation}, we use the LoopPoint module to show that this overhead makes analysis impractical for realistic input sizes.
These observations motivate the Nugget framework (Section~\ref{sec:the_nugget_framework}), which validates samples on real hardware without simulation and defines samples at the LLVM IR level so that the same set of representative regions can be used across different ISAs.
